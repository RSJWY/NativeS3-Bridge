# 多节点 mTLS 控制面与节点部署 — 技术设计

> 本文件是 PRD 的技术落地方案。控制协议已选定为 **WebSocket + JSON over mTLS**（见 PRD Open Questions）。
> 本任务仍处于规划阶段，本文不含可执行代码，只定义边界、契约、数据流与兼容/回滚形态。

---

## 1. 架构总览与职责边界

### 1.1 三类可交付物

单仓库,采用**硬切换**:废弃现有单体入口,拆分为面板与节点两个入口 + 一个共享协议包。现有部署必须整体切换到 panel+node,不再保留 standalone 一体化运行模式。

```
cmd/panel/            新增：面板进程入口（国内，集中管理面）
cmd/node/             新增：节点 Agent 入口（海外，S3 数据面 + 管理长连接客户端）
pkg/controlproto/     新增：面板/节点共享的控制协议定义（消息信封、类型、版本常量）
pkg/panel/            新增：面板业务（节点注册、PKI、期望状态、任务编排、审计）
pkg/nodeagent/        新增：节点 Agent（拨号、心跳、重连、期望状态对账、任务执行）
```

**硬切换取舍**:`cmd/natives3bridge/` 单体入口在本次演进后不再作为受支持的运行形态(可保留源码用于本地开发/测试,但不再作为部署目标,也不承诺其后续兼容)。回滚需整体回退到升级前的二进制 + 数据快照,而非在 standalone 与 panel+node 之间来回切换。这换来更干净的架构与单一权威模型;代价是升级不留"旧模式"退路,因此迁移流程本身必须足够稳(见 §8.3),把"平稳"落在迁移编排上而非双模式并存上。

现有 `pkg/server`(S3)、`pkg/storage`、`pkg/auth`、`pkg/quota`、`pkg/hooks`、`pkg/db`、`pkg/config` 继续被节点复用。`pkg/webadmin` 的**业务处理逻辑**（credentials/buckets/quota/reconcile/logs/dashboard 查询）由面板复用或迁移到面板侧,其 **HTTP 监听服务不在节点上启动**。

### 1.2 职责矩阵

| 能力 | 面板 (`cmd/panel`) | 节点 (`cmd/node`) | 共享 (`pkg/controlproto`) |
|---|---|---|---|
| WebAdmin UI / 管理 REST / 登录会话 | ✅ 唯一承载 | ❌ 不监听任何管理端口 | — |
| S3 数据面（监听、签名校验、对象读写） | ❌ | ✅ 唯一承载 | — |
| 节点注册 / 令牌签发 / PKI 生命周期 | ✅ | 仅本地生成私钥+CSR | — |
| 业务配置（凭据/Bucket/配额/ACL/Webhook/限流）权威 | ✅ 唯一权威 | 持久化并执行下发的期望状态 | 期望状态 schema |
| 期望状态版本管理与对账 | ✅ 生成单调版本 | 记录已应用版本、重连对账 | 版本号与对账消息 |
| 一次性任务（日志查询/扫描/校正） | ✅ 发起与结果聚合 | ✅ 执行 | 任务/回执消息 |
| 审计日志 | ✅ 集中记录 | 本地运行日志（有限） | — |
| S3 Secret Key 加密存储 | ✅ 加密主密钥与 DB 分离 | 明文仅存本地 DB 供 S3 校验 | — |

### 1.3 网络与端口

- 节点**只主动拨号**面板 `wss://panel:PORT/agent`(mTLS),面板不反向连接节点。
- 节点监听端口只有 S3 数据面(`server.s3_addr`),**不监听** admin/metrics/管理 API。
- 面板监听:①管理 UI/REST(沿用 `admin_addr`,人类管理员访问);②节点接入端点(mTLS WebSocket,独立监听地址与端口)。两者物理分离,便于分别做证书与防火墙策略。

## 2. 所有权与同步规则(凭据 / Bucket / 配额 / 统计 / 配置)

### 2.1 权威与副本

- **期望状态(desired state)**:凭据(不含明文 Secret Key 校验值以外的策略)、Bucket、ACL、配额、Webhook、限流策略——**面板是唯一权威**。节点持有的是"最后一次成功应用的副本"。
- **运行数据(observed state)**:对象、UsedBytes、RequestStat、本地运行日志——**节点是权威**,面板只做只读汇总(按需拉取或随心跳上报摘要)。
- **对象命名空间**:节点级独立,面板不承诺跨节点一致,不做跨节点复制。

### 2.2 期望状态版本

- 每个节点在面板侧有一条 `desired_config` 记录,含单调递增 `version`(int64)和内容哈希。
- 管理员每次变更 → 生成新 `version`,只保留**最新期望状态**,不逐条积压被覆盖的中间版本(PRD 明确)。
- 节点持久化 `applied_version`。重连时上报 `applied_version` + 本地内容哈希,面板比对:
  - `applied_version == desired.version && hash 一致` → `synced`
  - `applied_version < desired.version` → `waiting-sync`,面板下发最新期望状态
  - `hash 不一致但 version 相等` → `drift`(节点被本地直改数据库),标记漂移并重新对账
- 应用结果分节点记录:`synced` / `failed`(附错误)/ `drift`;批量操作**分节点记录结果,不把部分成功报告为全局成功**。

### 2.3 Secret Key 处理(核心安全约束)

- 面板 `NodeCredential` 表存储 `secret_key_ciphertext`(用主密钥 AEAD 加密),**主密钥与 DB 分离保存**(独立文件/KMS,通过 `panel` 配置指向)。
- Secret Key 只在**创建时**明文返回管理员一次;列表/详情/日志/审计接口一律不返回明文。更换走**轮换**(生成新 Secret Key,下发新期望状态,旧值失效)。
- 下发到节点:面板在下发期望状态时,通过已建立的 mTLS 通道把解密后的 Secret Key 明文随该凭据一起下发;节点本地 DB 存明文(S3 SigV4 校验需要),这是节点级既有行为,不额外上传回面板。
- 迁移导入:现有节点明文 Secret Key 仅可经已建立的 mTLS 通道上报,面板收到后立即加密存储。

### 2.4 现有 schema 的演进

现有 `pkg/db/models.go` 的 `Credential/Bucket/RequestStat/HookConfig` 是**节点级**表,保持不变(节点继续用)。面板侧新增独立 schema(见 §6),不复用节点表结构。

## 3. PKI 与节点身份生命周期

### 3.1 CA 层级

- **离线根 CA**:根私钥加密离线保存,不在面板日常路径上;只用于签发/轮换中间 CA。
- **在线中间 CA**:面板日常签发/续期节点客户端证书只用中间 CA。中间 CA 私钥属于面板恢复集(见 §7)。
- 节点接入端点服务器证书:由中间 CA(或独立服务器 CA)签发,供节点首次注册时做**服务器 TLS 校验面板身份**。

### 3.2 注册流程(单次令牌 → mTLS)

```
1. 管理员在面板创建逻辑节点 → 面板生成高强度、单次使用、默认 10 分钟有效的注册令牌
2. 节点首启：本机生成私钥（永不上传），生成 CSR
3. 节点用普通服务器 TLS 校验面板身份，POST 注册端点：{registration_token, csr}
4. 面板校验令牌（有效期内、未使用、匹配该逻辑节点）→ 中间 CA 签发客户端证书
   → 令牌立即失效
5. 节点保存签发的证书；此后所有常规管理连接必须用 mTLS，不得再用注册令牌
```

- 注册端点是面板上**独立于 mTLS 长连接**的一次性 HTTPS 端点(服务器 TLS,非 mTLS),仅接受令牌+CSR。
- 私钥保存在节点本地(`node.cert.key_file`),面板只存证书与指纹,不存私钥。

### 3.3 续期与轮换

- **续期**:证书临近到期,节点经 mTLS 通道用现证书请求续期,中间 CA 签发新证书,不需要新令牌。
- **轮换/重装**:节点重装或替换保留原逻辑节点、期望配置和历史归属;吊销旧证书,管理员签发新的单次令牌,节点重新走注册流程用新证书接入。逻辑节点内部唯一 ID **不复用**。

### 3.4 吊销与生命周期

- 节点生命周期:`active` / `disabled` / `retired`。
  - `disabled`:可恢复,暂停下发与任务,不吊销证书(或临时拒绝连接)。
  - `retired`:永久生效,**吊销全部证书与注册令牌**,UI"删除"实现为退役(保留审计关系,非物理删除)。
- 吊销校验:面板在 mTLS 握手后校验节点证书指纹是否在有效集合(维护吊销列表/有效指纹表),吊销的证书拒绝接入。
- **吊销只切断控制面,不停止节点 S3 数据面**;安全处置文档必须说明还需停止宿主机服务或轮换相关 S3 凭据。

## 4. 控制协议(WebSocket + JSON over mTLS)

### 4.1 连接建立

1. 节点用客户端证书拨号 `wss://panel:PORT/agent`,面板用中间 CA 校验客户端证书(mTLS)。
2. 握手后首帧 `hello`:节点上报 `protocol_version`、`node_id`、`applied_version`、本地内容哈希、Agent 版本。
3. 面板回 `hello_ack`:协商后的 `protocol_version`、服务器时间(供时钟偏差检测)、是否需要对账。

### 4.2 消息信封

所有消息统一信封,`type` 分发,`id` 用于任务幂等与回执关联:

```json
{
  "type": "hello | hello_ack | heartbeat | heartbeat_ack | desired_state | ack | task | task_result | error",
  "version": 1,
  "id": "uuid",          // 任务/请求关联 ID；心跳可省略
  "ts": "RFC3339",
  "payload": { ... }     // 按 type 定义的结构
}
```

### 4.3 消息类型

| type | 方向 | 用途 |
|---|---|---|
| `hello` / `hello_ack` | 节点→面板 / 面板→节点 | 版本协商、身份、对账触发、时钟同步 |
| `heartbeat` / `heartbeat_ack` | 双向 | 保活、上报在线状态与运行摘要 |
| `desired_state` | 面板→节点 | 下发最新期望状态(全量,含 version) |
| `ack` | 节点→面板 | 期望状态应用结果(synced/failed/drift + version) |
| `task` | 面板→节点 | 一次性任务(日志查询/扫描/校正预览/校正应用) |
| `task_result` | 节点→面板 | 任务结果回执(含 task id、状态、受限结果集) |
| `error` | 双向 | 协议级错误(版本不兼容、鉴权失败等) |

### 4.4 版本协商

- `pkg/controlproto` 定义 `ProtocolVersion` 常量与最小兼容版本。
- `hello`/`hello_ack` 交换版本;不兼容时面板回 `error` 并关闭连接,节点记录并告警(不无限重连打洞)。
- 面板镜像与节点镜像可独立构建/升级/回滚,但共享有版本约束的 `pkg/controlproto`;新增消息字段向后兼容(未知字段忽略),破坏性变更升 `ProtocolVersion`。

### 4.5 心跳、重连、超时、背压

- **心跳**:节点周期发 `heartbeat`(默认 15s),面板 `heartbeat_ack`。面板超过 `N × 间隔`(默认 3 倍)未收到 → 标记节点 `offline`。
- **重连**:节点断线后指数退避重连(带上限与抖动);重连成功走 §4.1 对账。
- **超时**:每类任务有独立超时;`desired_state` 下发需在超时内收到 `ack`,否则标记 `waiting-sync/failed`。
- **背压**:
  - 单连接**未确认任务数上限**(在途 task 数),超限则面板暂缓下发新任务。
  - **消息大小上限** + **日志查询结果条数/时间范围/响应字节上限**——控制连接不作无界日志流。
  - 节点侧任务并发上限,超出排队或拒绝(高风险任务不排队等未来自动执行,见 §5)。

## 5. 任务下发规则(幂等 / 确认 / 过期 / 失败 / 离线)

### 5.1 任务模型

- 任务受限于**预定义操作**:`log_query`、`storage_scan`(只读预览)、`storage_reconcile_apply`(实际应用)。**不提供任意远程 Shell**,不预留通用命令通道。
- 每个任务有 `task_id`(幂等键)、`type`、`params`、`timeout`、`created_by`(统一管理员身份)、`target_node`。

### 5.2 幂等与确认

- 节点按 `task_id` 去重:重复收到同一 `task_id` 只执行一次,重发 `task_result`。
- 任务完成回 `task_result`,面板记录操作者、目标节点、参数、结果(扫描/校正必须区分 preview 与 apply)。

### 5.3 过期、失败、离线

- **只向在线节点发起**一次性任务;节点离线**不排队**、不等待未来自动执行(PRD 明确)。
- 任务发出后连接中断 → 面板标记该任务 `failed` 或 `unknown`,**要求管理员重新确认**,高风险操作(校正 apply)**不静默重试**。
- 期望状态(非一次性任务)与一次性任务区别对待:期望状态离线时保存最新版本待重连对账;一次性任务离线时直接拒绝发起。

## 6. 面板数据模型(新增,独立于节点表)

面板侧 GORM 模型(草案,最终字段在实现期确定):

| 表 | 关键字段 | 说明 |
|---|---|---|
| `nodes` | `id`(内部唯一,不复用)、`display_name`(可改)、`status`(active/disabled/retired)、`created_at`、`retired_at` | 逻辑节点 |
| `node_certs` | `node_id`、`fingerprint`、`serial`、`not_before/after`、`revoked`、`revoked_at` | 证书生命周期,支持吊销 |
| `registration_tokens` | `node_id`、`token_hash`、`expires_at`、`used_at` | 单次令牌,存哈希不存明文 |
| `desired_configs` | `node_id`、`version`(单调)、`content_json`、`content_hash`、`updated_by`、`updated_at` | 只存最新期望状态 |
| `node_status` | `node_id`、`online`、`applied_version`、`sync_state`(synced/waiting/failed/drift)、`last_heartbeat` | 观测状态 |
| `node_credentials` | `node_id`、`access_key`、`secret_key_ciphertext`、策略字段 | Secret Key 加密,主密钥外置 |
| `tasks` | `task_id`、`node_id`、`type`、`params`、`state`、`result_json`、`created_by`、`created_at` | 一次性任务及回执 |
| `audit_logs` | `ts`、`action`、`target_node`、`target_resource`、`result`、`source` | 见 §7,不含敏感明文 |

- 主密钥:面板配置指向独立文件/KMS(不落 DB);`node_credentials.secret_key_ciphertext` 用它加解密。
- 迁移:面板 DB 用独立迁移注册表(仿 `pkg/db/migrate.go` 的 `migrationModels` + 校验模式),与节点 DB 迁移互不干扰。

## 7. 故障、审计、安全边界

### 7.1 故障行为

| 场景 | 行为 |
|---|---|
| 节点失联(面板视角) | 标记 offline,暂停下发与任务;不影响该节点 S3 数据面 |
| 面板不可用(节点视角) | 节点继续用最后应用的持久化配置提供 S3;不接受任何本地新业务配置变更;恢复后自动重连+对账 |
| 面板 DB 故障 | 管理面暂停;节点数据面不受影响;恢复后对账 |
| 证书过期 | 到期前主动续期;已过期则拒绝接入,需管理员处理(续期或重签) |
| 时钟偏差 | `hello_ack`/心跳携带服务器时间,节点检测偏差并告警;令牌有效期以面板时钟为准 |
| 数据库直改(紧急人工修复) | 通过内容哈希检测为 `drift`,重连时重新对账 |

- **失败关闭(fail-closed)**:CA 或 Secret Key 加密主密钥确实丢失时,接受重新注册节点或轮换 S3 凭据,**不提供绕过 mTLS 或导出明文 Secret Key 的后门**。

### 7.2 审计

- 审计日志至少记录:时间、操作类型、目标节点、目标资源、结果、请求来源。
- **绝不记录**:私钥、完整 S3 Secret Key、注册令牌明文、Session 凭据。
- 所有管理操作(含批量、任务)写审计,记录统一管理员身份。

### 7.3 备份与恢复集

完整恢复集必须覆盖:面板 DB、Secret Key 加密主密钥、根 CA 证书与加密根私钥、在线中间 CA 证书与私钥、面板配置、必要审计数据。

- **DB 备份与主密钥分别备份**:仅拿到 DB 备份不得直接恢复出明文 Secret Key。
- 恢复面板**不应要求重新注册**未被吊销且证书仍有效的节点(证书指纹表随 DB 恢复,节点自动重连即可)。

## 8. 镜像、配置、部署、升级、兼容、回滚

### 8.1 两个镜像

- **面板镜像**:`cmd/panel`,内嵌 WebAdmin UI(现有 `pkg/webadmin/ui` 前端资源迁移到面板),暴露管理 UI 端口 + 节点接入端口。
- **节点镜像**:`cmd/node`,保留 S3/存储/数据面,**不启动 WebAdmin 监听**,不嵌入或不暴露管理 UI。
- 两镜像独立构建/升级/回滚,共享 `pkg/controlproto`(有版本约束)。现有 `Dockerfile` 拆分或参数化为两个构建目标。

### 8.2 节点本地配置(仅基础设施)

节点 `config.yaml` 只保留启动必需项:面板地址、节点证书/私钥/CA 路径、S3 监听地址、数据目录、DB 连接、S3 TLS。业务配置(凭据/Bucket/配额/Webhook/限流)不再由节点本地配置管理,改由期望状态下发。

### 8.3 单节点原地升级(硬切换,不留退路)

升级姿态为**硬切换**:不再维护单体/standalone 入口,现有用户必须同时部署面板 + node 镜像。因此"平稳"不依赖保留旧二进制,而由下面的迁移流程与三条安全网保证升级期间 **S3 数据面不中断**。

```
1. 现有单节点升级为 node 镜像，保留对象目录与节点数据库（同一 data_root 与 DB）
2. node 启动即用本地已有 DB 照常提供 S3 服务（见安全网 A），不等待面板
3. 管理员在面板创建逻辑节点 + 一次性令牌，节点完成 mTLS 注册
4. 节点只读上报现有 Bucket/凭据/配额/当前状态（含明文 Secret Key 经 mTLS 上报）
5. 面板展示导入摘要 → 管理员确认
6. 确认前不向节点覆盖任何业务配置；确认后生成初始期望状态基线（version=1），切换为面板权威
7. 迁移不搬迁对象文件、不迁历史日志
```

**安全网 A — 未注册初始态照常服务 S3。**
node 的 S3 数据面启动**不以面板注册为前提**。区分两种"面板不可达"语义并都要照常提供 S3:
- **从未注册(初始态)**:node 直接使用本地已有 DB(升级前单体写入的 credentials/buckets/quota)提供 S3,后台并行尝试注册。这消除了"换镜像后到注册完成前 S3 短暂不可用"的空窗。
- **注册过但当前失联**:使用最后一次成功应用并持久化的期望状态继续服务(§7 已覆盖)。
两态都不允许通过节点管理接口产生新的业务配置变更。

**安全网 B — node 忽略不再管理的旧配置字段。**
现有 `config.yaml` 含业务配置相关字段(升级前经 WebAdmin/sed 落库)。node 入口对这些"不再由节点管理"的字段必须**忽略而非报错**,只消费 §8.2 的基础设施字段。避免现有配置文件直接导致 node 启动失败。校验只针对基础设施字段生效。

**安全网 C — 节点 DB 迁移严格增量。**
node 新增的表(已应用期望状态版本、漂移状态、任务执行记录等)必须是**只增表/只增列,不改不删** credentials/buckets/request_stats 等现有结构。这样:①升级迁移不触碰存量数据;②回滚到单体二进制时,旧代码忽略新增表即可继续跑(见 §8.4)。迁移沿用 `pkg/db/migrate.go` 的 `AutoMigrate` + `validateSchema` 模式,新表加入注册表与期望表/索引校验清单。

- **迁移失败**:保持节点原有 S3 服务与本地状态可用,允许管理员安全重试或放弃接管(放弃则 node 回到"未注册但正常服务 S3"的初始态,等待重新发起注册)。

### 8.4 回滚

- **协议/镜像层面**:面板/节点镜像各自可回滚到上一版本;`pkg/controlproto` 版本兼容保证新旧混跑期间协议可协商或明确拒绝。
- **硬切换的整体回退**:由于不保留单体入口,回滚到多节点方案之前的版本 = 把 node 换回旧单体二进制。安全网 C(增量迁移)保证这条路可用:旧单体二进制忽略 node 新增的表,直接用未被改动的 credentials/buckets 继续跑。前提是回退前已停止面板对该节点的期望状态下发,避免旧二进制读到面板改写过的业务数据后语义不一致——回退属人工处置动作,须在面板侧先停用(disable)该节点。
- **期望状态层面**:期望状态有版本号,误下发可由管理员发布新版本纠正(不做自动回滚,避免与漂移检测冲突)。

## 9. 关键取舍

- **WebSocket+JSON 而非 gRPC**:复用现有 `net/http` 栈,零 protobuf 工具链,JSON 可读利于跨境调试与审计;背压/幂等本就要手写,gRPC 收益有限。第一版节点规模小,无需 HTTP/2 高吞吐流控。
- **面板复用 WebAdmin 业务逻辑而非重写**:降低第一版工作量;节点侧仅去除 HTTP 监听,业务处理下沉为期望状态执行器。
- **只存最新期望状态**:避免命令积压与重放复杂度;代价是失去中间版本历史(审计日志补足变更记录)。
- **Secret Key 明文下发到节点**:S3 SigV4 校验的必然要求;通过 mTLS 通道 + 面板加密存储把风险限制在传输与面板存储两端。
